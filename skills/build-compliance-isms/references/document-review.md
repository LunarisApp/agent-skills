# Guided document review

## Before questions

Read document, related findings, sources, services, TBDs, issues, risks, and linked documents. Inspect repository for discoverable facts first. Do not ask user what source already answers.

## Question design

Ask one compact batch focused on decisions that materially change document. Explain consequences and examples when terminology is unclear. Do not recommend an arbitrary operational value. User may answer, skip, pause, or mark TBD.

Separate these states in wording:

- Observed current practice.
- User-confirmed current practice.
- Explicit policy commitment.
- Planned improvement.
- Unknown or undecided.

## After answers

1. Update document with accurate ISMS language.
2. Add unresolved decisions to `TBD.md` only after initial document input.
3. Add confirmed deficiencies to `ISSUES.md` with source and owner still user-defined when unknown.
4. Add recurring actions to `SCHEDULE.md` only when owner and cadence are established.
5. Update `STATUS.md` workflow stage and counts.
6. Update `STATE.md` with next action.
7. Ask for explicit approval only when document is consistent and blocking TBDs are resolved.

## Wording rules

Use “must” only for approved commitment, “currently” only for verified practice, and “planned” for future work. Avoid certification claims, absolute security claims, and statements that evidence exists without verification.
